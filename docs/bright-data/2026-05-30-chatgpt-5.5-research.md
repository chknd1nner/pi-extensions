---

Yes. I’d build this as a **normal Pi TypeScript extension that registers two LLM-callable tools**, not as a Pi model provider. Bright Data is supplying external web context, not model inference.

The clean shape is:

`web_search` → Bright Data **SERP/Search API** → normalized Markdown SERP list + structured JSON in `details`.

`web_fetch` → Bright Data **Unlocker API** → cleaned Markdown page body + source metadata.

Bright Data’s Web Access docs split the product family into Unlocker API, Search API, Browser API, and Crawl API. For your use case, Search/SERP is the right primitive for query-to-results, and Unlocker is the right primitive for arbitrary URL-to-page-content. Bright Data says Unlocker handles blocking, IP rotation, browser fingerprints, sessions, and CAPTCHA/challenge resolution; Search API returns structured real-time search engine data such as rankings, organic links, ads, and other SERP elements. ([docs.brightdata.com][1])

Pi-side, this is exactly what extensions are for: Pi is deliberately small at the core and pushes workflow-specific behavior into extensions, skills, prompt templates, themes, and packages. Extensions can register custom tools callable by the LLM via `pi.registerTool()`, and they can be loaded project-locally from `.pi/extensions/` or tested directly with `pi -e`.   

## The Bright Data API shape

Both APIs use the same central endpoint:

```text
POST https://api.brightdata.com/request
Authorization: Bearer <BRIGHTDATA_API_KEY>
Content-Type: application/json
```

For SERP, the docs show:

```json
{
  "zone": "serp_api1",
  "url": "https://www.google.com/search?q=pizza",
  "format": "json"
}
```

The response includes structured search metadata and results, and Bright Data documents `zone`, `url`, and `format` as required body fields. `format: "json"` returns structured data; `format: "raw"` returns raw HTML. ([docs.brightdata.com][2]) ([docs.brightdata.com][2])

For Unlocker, the docs show the same `/request` endpoint with a Web Unlocker zone:

```json
{
  "zone": "web_unlocker1",
  "url": "https://geo.brdtest.com/welcome.txt",
  "format": "raw"
}
```

Unlocker also supports `data_format: "markdown"` to transform HTML into clean Markdown, which is exactly what you want for LLM consumption. The docs also state the URL must include `http` or `https` and be publicly accessible. ([docs.brightdata.com][3]) ([docs.brightdata.com][3])

Bright Data’s own MCP docs are a useful design reference too: their Rapid/Free mode exposes tools conceptually equivalent to this—search results and single-page scraping as clean Markdown—while batch search/fetch is listed under more advanced tooling. That suggests your two-tool Pi extension is aligned with their own agent-facing abstraction. ([docs.brightdata.com][4])

## Core helper: one Bright Data request function

This is the key abstraction. Both tools should call the same function.

```ts
// .pi/extensions/brightdata/brightdata.ts
type BrightDataRequest = {
  zone: string;
  url: string;
  format: "raw" | "json";
  method?: "GET" | "POST";
  country?: string;
  data_format?: "markdown" | "screenshot";
  body?: string;
};

type BrightDataResponse = {
  status_code?: number;
  headers?: Record<string, string>;
  body?: unknown;
  [key: string]: unknown;
};

export async function brightDataRequest(
  payload: BrightDataRequest,
  signal?: AbortSignal,
): Promise<unknown> {
  const apiKey = process.env.BRIGHTDATA_API_KEY;
  if (!apiKey) {
    throw new Error("BRIGHTDATA_API_KEY is not set.");
  }

  const response = await fetch("https://api.brightdata.com/request", {
    method: "POST",
    signal,
    headers: {
      "Authorization": `Bearer ${apiKey}`,
      "Content-Type": "application/json",
    },
    body: JSON.stringify(payload),
  });

  const text = await response.text();

  let parsed: unknown = text;
  const contentType = response.headers.get("content-type") ?? "";

  if (contentType.includes("application/json") || text.trim().startsWith("{")) {
    try {
      parsed = JSON.parse(text);
    } catch {
      // Keep text fallback. Bright Data can return raw bodies depending on format.
    }
  }

  if (!response.ok) {
    const message =
      typeof parsed === "object" && parsed !== null
        ? JSON.stringify(parsed)
        : String(parsed);

    throw new Error(`Bright Data request failed: HTTP ${response.status}: ${message}`);
  }

  return parsed;
}

export function responseBodyToText(value: unknown): string {
  if (typeof value === "string") return value;

  const maybe = value as BrightDataResponse;
  if (typeof maybe?.body === "string") return maybe.body;

  if (maybe?.body !== undefined) return JSON.stringify(maybe.body, null, 2);

  return JSON.stringify(value, null, 2);
}
```

The important choices here are: use `fetch(..., { signal })` so Pi’s abort/escape can cancel the request, parse JSON opportunistically, and normalize Bright Data’s common `{ status_code, headers, body }` wrapper into text when needed.

## URL validation for `web_fetch`

Because `web_fetch` accepts arbitrary URLs, add a small guard. Bright Data says target URLs must be public `http` or `https` URLs, so the tool should reject obvious local/private targets before spending API credits. ([docs.brightdata.com][3])

```ts
// .pi/extensions/brightdata/normalize.ts
const PRIVATE_HOSTS = new Set([
  "localhost",
  "127.0.0.1",
  "0.0.0.0",
  "::1",
]);

export function assertPublicHttpUrl(raw: string): string {
  let url: URL;

  try {
    url = new URL(raw);
  } catch {
    throw new Error(`Invalid URL: ${raw}`);
  }

  if (url.protocol !== "http:" && url.protocol !== "https:") {
    throw new Error(`Only http/https URLs are supported: ${raw}`);
  }

  const host = url.hostname.toLowerCase();

  if (PRIVATE_HOSTS.has(host)) {
    throw new Error(`Refusing to fetch local/private host: ${raw}`);
  }

  if (
    host.startsWith("10.") ||
    host.startsWith("192.168.") ||
    /^172\.(1[6-9]|2\d|3[0-1])\./.test(host)
  ) {
    throw new Error(`Refusing to fetch private network URL: ${raw}`);
  }

  return url.toString();
}

export function truncate(text: string, maxChars: number): string {
  if (text.length <= maxChars) return text;
  return text.slice(0, maxChars) + `\n\n[Truncated at ${maxChars} characters]`;
}
```

This is not perfect SSRF protection because DNS can still resolve weirdly, but it blocks the common footguns. Since Bright Data is fetching from their infrastructure, the risk is less “your LAN gets hit” and more “the tool is used carelessly,” but validation is still cheap and worthwhile.

## SERP normalizer

Bright Data’s structured SERP payloads may vary by engine and SERP feature. Don’t make your extension brittle by assuming only one exact shape. Normalize organic-ish results into a small schema.

```ts
type SearchResult = {
  rank?: number;
  title: string;
  url: string;
  snippet?: string;
  source?: string;
};

function pickString(...values: unknown[]): string | undefined {
  for (const value of values) {
    if (typeof value === "string" && value.trim()) return value.trim();
  }
  return undefined;
}

function normalizeSearchResults(payload: unknown): SearchResult[] {
  const root = payload as any;

  const candidates =
    root?.organic ??
    root?.organic_results ??
    root?.results ??
    root?.items ??
    root?.body?.organic ??
    root?.body?.organic_results ??
    [];

  if (!Array.isArray(candidates)) return [];

  return candidates
    .map((item: any, index: number): SearchResult | null => {
      const url = pickString(item.link, item.url, item.href);
      const title = pickString(item.title, item.name);
      const snippet = pickString(item.description, item.snippet, item.text);

      if (!url || !title) return null;

      return {
        rank: typeof item.rank === "number" ? item.rank : index + 1,
        title,
        url,
        snippet,
        source: pickString(item.source, item.displayed_link, item.domain),
      };
    })
    .filter((x): x is SearchResult => Boolean(x));
}

function formatSerpMarkdown(query: string, results: SearchResult[]): string {
  if (results.length === 0) {
    return `No normalized organic results found for: ${query}`;
  }

  return [
    `Search results for: ${query}`,
    "",
    ...results.map((result) => {
      const parts = [
        `${result.rank ?? "-"}\\. ${result.title}`,
        `   URL: ${result.url}`,
      ];

      if (result.snippet) parts.push(`   Snippet: ${result.snippet}`);
      if (result.source) parts.push(`   Source: ${result.source}`);

      return parts.join("\n");
    }),
  ].join("\n");
}
```

The normalized Markdown is what you put in the visible tool result. Keep the raw Bright Data response in `details` so a later renderer/debugger can inspect it without forcing the LLM to consume a giant JSON blob.

## The Pi extension: `web_search` and `web_fetch`

Pi docs show the extension pattern: export a default function receiving `ExtensionAPI`, import `Type` from `typebox`, and call `pi.registerTool()` with `name`, `description`, `parameters`, and `execute`. Pi also supports `promptSnippet` and `promptGuidelines`; use them so the model knows when to call these tools. 

```ts
// .pi/extensions/brightdata/index.ts
import type { ExtensionAPI } from "@earendil-works/pi-coding-agent";
import { Type, type Static } from "typebox";

import { brightDataRequest, responseBodyToText } from "./brightdata.js";
import { assertPublicHttpUrl, truncate } from "./normalize.js";

const SearchSchema = Type.Object({
  queries: Type.Array(Type.String({ minLength: 1 }), {
    minItems: 1,
    maxItems: 10,
    description: "One or more web search queries.",
  }),
  engine: Type.Optional(
    Type.Union([
      Type.Literal("google"),
      Type.Literal("bing"),
      Type.Literal("duckduckgo"),
      Type.Literal("yandex"),
    ], {
      description: "Search engine to target. Defaults to google.",
    }),
  ),
  country: Type.Optional(Type.String({
    description: "Two-letter country code, e.g. au, us, gb.",
  })),
  language: Type.Optional(Type.String({
    description: "Language code, e.g. en.",
  })),
  limit: Type.Optional(Type.Number({
    minimum: 1,
    maximum: 20,
    description: "Max normalized results per query. Defaults to 10.",
  })),
});

type SearchParams = Static<typeof SearchSchema>;

const FetchSchema = Type.Object({
  urls: Type.Array(Type.String({ minLength: 1 }), {
    minItems: 1,
    maxItems: 10,
    description: "One or more public http/https URLs to fetch.",
  }),
  country: Type.Optional(Type.String({
    description: "Two-letter country code, e.g. au, us, gb.",
  })),
  max_chars_per_page: Type.Optional(Type.Number({
    minimum: 1000,
    maximum: 100000,
    description: "Truncate each fetched page to this many characters. Defaults to 30000.",
  })),
});

type FetchParams = Static<typeof FetchSchema>;

function getRequiredEnv(name: string): string {
  const value = process.env[name];
  if (!value) throw new Error(`${name} is not set.`);
  return value;
}

function makeSearchUrl(
  engine: SearchParams["engine"] = "google",
  query: string,
  language = "en",
): string {
  const q = encodeURIComponent(query);

  switch (engine) {
    case "bing":
      return `https://www.bing.com/search?q=${q}`;
    case "duckduckgo":
      return `https://duckduckgo.com/html/?q=${q}`;
    case "yandex":
      return `https://yandex.com/search/?text=${q}`;
    case "google":
    default:
      return `https://www.google.com/search?q=${q}&hl=${encodeURIComponent(language)}`;
  }
}

type SearchResult = {
  rank?: number;
  title: string;
  url: string;
  snippet?: string;
  source?: string;
};

function pickString(...values: unknown[]): string | undefined {
  for (const value of values) {
    if (typeof value === "string" && value.trim()) return value.trim();
  }
  return undefined;
}

function normalizeSearchResults(payload: unknown): SearchResult[] {
  const root = payload as any;

  const candidates =
    root?.organic ??
    root?.organic_results ??
    root?.results ??
    root?.items ??
    root?.body?.organic ??
    root?.body?.organic_results ??
    [];

  if (!Array.isArray(candidates)) return [];

  return candidates
    .map((item: any, index: number): SearchResult | null => {
      const url = pickString(item.link, item.url, item.href);
      const title = pickString(item.title, item.name);
      const snippet = pickString(item.description, item.snippet, item.text);

      if (!url || !title) return null;

      return {
        rank: typeof item.rank === "number" ? item.rank : index + 1,
        title,
        url,
        snippet,
        source: pickString(item.source, item.displayed_link, item.domain),
      };
    })
    .filter((item): item is SearchResult => Boolean(item));
}

function formatSearchBlock(query: string, results: SearchResult[]): string {
  if (results.length === 0) {
    return `## ${query}\n\nNo normalized organic results found.`;
  }

  return [
    `## ${query}`,
    "",
    ...results.map((result) => {
      return [
        `${result.rank ?? "-"}\\. ${result.title}`,
        `   URL: ${result.url}`,
        result.snippet ? `   Snippet: ${result.snippet}` : undefined,
        result.source ? `   Source: ${result.source}` : undefined,
      ].filter(Boolean).join("\n");
    }),
  ].join("\n");
}

async function mapLimit<T, R>(
  items: T[],
  limit: number,
  fn: (item: T, index: number) => Promise<R>,
): Promise<R[]> {
  const results: R[] = [];
  let next = 0;

  async function worker() {
    while (next < items.length) {
      const index = next++;
      results[index] = await fn(items[index], index);
    }
  }

  await Promise.all(
    Array.from({ length: Math.min(limit, items.length) }, () => worker()),
  );

  return results;
}

export default function brightDataExtension(pi: ExtensionAPI) {
  pi.registerTool({
    name: "web_search",
    label: "Web Search",
    description:
      "Search the web using Bright Data SERP API and return a concise, normalized list of results.",
    promptSnippet:
      "Search the live web with Bright Data SERP API and return ranked result URLs/snippets.",
    promptGuidelines: [
      "Use web_search when the user asks for current, recent, niche, or externally verifiable information.",
      "Use web_search before web_fetch when you need to discover relevant URLs.",
      "Do not use web_search for local repository files; use Pi file tools for that.",
    ],
    parameters: SearchSchema,

    async execute(_toolCallId, params: SearchParams, signal) {
      const zone = getRequiredEnv("BRIGHTDATA_SERP_ZONE");
      const country = params.country ?? process.env.BRIGHTDATA_DEFAULT_COUNTRY;
      const limit = params.limit ?? 10;

      const perQuery = await mapLimit(params.queries, 3, async (query) => {
        const payload = await brightDataRequest({
          zone,
          url: makeSearchUrl(params.engine ?? "google", query, params.language ?? "en"),
          format: "json",
          country,
        }, signal);

        const results = normalizeSearchResults(payload).slice(0, limit);

        return {
          query,
          results,
          raw: payload,
        };
      });

      const markdown = perQuery
        .map((entry) => formatSearchBlock(entry.query, entry.results))
        .join("\n\n---\n\n");

      return {
        content: [{ type: "text", text: markdown }],
        details: {
          provider: "brightdata",
          tool: "web_search",
          engine: params.engine ?? "google",
          country,
          queries: perQuery,
        },
      };
    },
  });

  pi.registerTool({
    name: "web_fetch",
    label: "Web Fetch",
    description:
      "Fetch public web pages through Bright Data Web Unlocker and return cleaned Markdown.",
    promptSnippet:
      "Fetch public URLs through Bright Data Unlocker and return clean Markdown page content.",
    promptGuidelines: [
      "Use web_fetch when the user provides URLs or after web_search finds pages that need reading.",
      "Use web_fetch for public http/https URLs only.",
      "Prefer a small number of high-value URLs rather than fetching many pages unnecessarily.",
    ],
    parameters: FetchSchema,

    async execute(_toolCallId, params: FetchParams, signal) {
      const zone = getRequiredEnv("BRIGHTDATA_UNLOCKER_ZONE");
      const country = params.country ?? process.env.BRIGHTDATA_DEFAULT_COUNTRY;
      const maxChars = params.max_chars_per_page ?? 30_000;

      const urls = params.urls.map(assertPublicHttpUrl);

      const pages = await mapLimit(urls, 3, async (url) => {
        const payload = await brightDataRequest({
          zone,
          url,
          format: "raw",
          data_format: "markdown",
          country,
        }, signal);

        const markdown = truncate(responseBodyToText(payload), maxChars);

        return {
          url,
          markdown,
          raw: payload,
        };
      });

      const text = pages.map((page, index) => {
        return [
          `# Page ${index + 1}: ${page.url}`,
          "",
          page.markdown,
        ].join("\n");
      }).join("\n\n---\n\n");

      return {
        content: [{ type: "text", text }],
        details: {
          provider: "brightdata",
          tool: "web_fetch",
          country,
          pages,
        },
      };
    },
  });
}
```

A few details matter here.

First, both tools cap batches at 10 and run with concurrency 3. That keeps the model from casually burning your free tier.

Second, `web_search` returns both readable Markdown and structured `details`. The LLM sees a clean ranked list, but you still have raw payloads available in the session/tool result for debugging.

Third, `web_fetch` asks Bright Data for `data_format: "markdown"`. That means you should not need your own Readability/JSDOM/html-to-md pipeline for the basic version. Bright Data documents `data_format: "markdown"` as the response transformation for clean Markdown. ([docs.brightdata.com][3])

Fourth, both tools pass Pi’s `signal` to `fetch`. That is worth doing. Escape/abort should cancel remote browsing work instead of leaving requests hanging.

## How the tools should appear to the model

I would describe them to the LLM roughly like this through `promptSnippet` and `promptGuidelines`:

`web_search` is for discovery. Use it when you need fresh/current/recent web information, or when the user asks for something not in the repo. It returns ranked URLs, titles, snippets, and optional raw SERP details.

`web_fetch` is for reading. Use it when the user gives URLs, or after `web_search` finds likely sources. It returns cleaned Markdown and source URLs.

Do not make `web_fetch` silently search. Keep search and fetch separate. That separation gives the agent better planning behavior: search broadly, inspect selectively, then answer.

## Handling multiple queries and URLs

For `web_search`, batching is useful because LLMs often need query reformulation:

```json
{
  "queries": [
    "Bright Data SERP API markdown data_format",
    "Bright Data Unlocker API data_format markdown",
    "Pi coding agent extension registerTool"
  ],
  "engine": "google",
  "country": "au",
  "limit": 5
}
```

For `web_fetch`, batching is useful after search:

```json
{
  "urls": [
    "https://docs.brightdata.com/scraping-automation/web-unlocker/send-your-first-request",
    "https://docs.brightdata.com/api-reference/rest-api/serp/serp-api"
  ],
  "max_chars_per_page": 30000
}
```

I’d resist adding an “auto-follow top N search results” option at first. It sounds convenient, but it encourages expensive, noisy fetch behavior. Let the agent explicitly decide which URLs matter.

## Error handling you want before calling this “done”

Your extension should turn Bright Data/API failures into clean tool errors:

```ts
function explainBrightDataError(error: unknown): string {
  const message = error instanceof Error ? error.message : String(error);

  if (message.includes("401")) {
    return "Bright Data authentication failed. Check BRIGHTDATA_API_KEY.";
  }

  if (message.includes("403")) {
    return "Bright Data rejected the request. Check zone permissions and product access.";
  }

  if (message.includes("429")) {
    return "Bright Data rate limit or quota was hit. Try fewer queries/URLs.";
  }

  return message;
}
```

Then inside each `execute`:

```ts
try {
  // real work
} catch (error) {
  return {
    isError: true,
    content: [{ type: "text", text: explainBrightDataError(error) }],
    details: { error: String(error) },
  };
}
```

That is better than throwing raw errors into the UI, especially for family/FamilyOS-style use.

## Optional: custom TUI rendering

For the first version, don’t bother. Plain Markdown result text is enough.

Later, add `renderResult` so search results show as compact clickable-looking rows in the terminal. Pi supports custom rendering for tool calls/results, and tool renderers return TUI `Component`s. 

The practical sequence should be:

1. Build plain text tools.
2. Verify Bright Data zones and free-tier limits.
3. Add markdown rendering polish.
4. Package it as `pi-brightdata` only after you use it for a week.

## One important correction to avoid a future refactor

Do not name this a “Bright Data provider” inside Pi, even though Bright Data calls itself a provider of data APIs. In Pi terms, “provider” usually means an LLM provider/model backend. This is an **extension** with **tools**.

A good package name would be something like:

```text
pi-brightdata-tools
```

Tool names:

```text
web_search
web_fetch
```

Env vars:

```text
BRIGHTDATA_API_KEY
BRIGHTDATA_SERP_ZONE
BRIGHTDATA_UNLOCKER_ZONE
BRIGHTDATA_DEFAULT_COUNTRY
```

That naming will keep it mentally compatible with Pi’s architecture and future-proof if you later add `web_screenshot`, `web_crawl`, or `browser_action`.
